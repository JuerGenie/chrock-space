## 前言

很早之前就有从头造一个博客轮子的想法，一直到最近才开始真正的着手做。关于技术选型上也没有多想，因为当时正在接触 svelte（用来编写 web component），索性便接着用它来写博客页面。

在最开始的时候，关于如何解析 markdown 文档，进行了一些思考，最初时打算自己使用 marked.js 或 markdown-it 来着手写一个解析模块，只是因为我太懒了（笑）所以写了一半就没再继续写下去。

因为最关键的文章解析没搞定，所以博客又这么搁置了一段时间，直到最近工作稍微轻松了一些，有了比较多的闲暇时间来研究。之所以会想到重新着手做博客，是因为无意中发现了 MDX 这种奇怪的东西，好奇之下试着用了用，感觉十分有趣。

> MDX 可以认为是 markdown 的超集，能够允许使用者在文档中直接编写 JSX 代码，并使其能够正确的被编译。
> 
> 更简单的说，MDX 使 markdown 能够作为组件来进行编写，也能与其他组件无缝的整合在一起，通过组件化，为 markdown 文档赋予更多的表现力。
> 
> ↑ 嗯，只是我个人的理解。

大概了解了这东西怎么时候后，又试着找了找有没有 svelte 下的替代品，这并不难找，在 [npmjs.cn](https://www.npmjs.cn) 中使用关键词进行搜索，结果很快便浮出水面：`svelte-preprocess-markdown`。

现在这篇文章就通过 `svelte-preprocess-markdown` 进行编译。

## 关于博客的一些设计思路的记录

作为一个轮子，从头开始设计一个博客页面是很有意思的事情，现在的 CHROCK-SPACE 依然只是个雏形，但在后续我会继续进行更新迭代，必要时也会推倒重来，因为只是个练手的项目，所以完全不会有什么额外的负担。

当前版本下，CHROCK-SPACE 使用 svelte 来编写页面，同时使用 `svelte-preprocess-markdown` 来将 markdown 文章编译为静态的 svelte 组件，并依赖于这一能力实现编写具有一定交互的文档的能力。

### 关于页面结构

关于博客的页面结构，大致如下：

```text
+============================+
| 博客标题              导航 |
+============================+
|   [] +==============+      |
|      |              |      |
|      |   页面内容   |      |
|      |              |      |
|   [] +==============+      |
+============================+
```

### 关于博客的子页面

关于博客的页面划分：

1. 首先有一个主页，用于展示最新的文章，以及一些公告，也可能不是博客的最新资讯，而仅仅是一个开屏信息，这个还要看之后的想法，目前就先这么着。
2. 接着要有一个文章页，用于阅读文章。
3. 再来，需要有一个文章列表页，用于列出所有文章，并提供查询功能。
    - 按文章分组
    - 按文章标签
    - 按文章标题
4. 最后要有一个关于我的介绍页，这大概可以用侧滑栏或模态框来做，而不一定是个子页面。

于是，这显然让博客变成了一个多页面应用，不过我不太希望将应用构建成多页面应用，这样**不太灵活**，每个页面都需要一份配置，就算是通过遍历去生成配置代码也很麻烦。所以我决定将博客构建为单页面应用（SPA），因此我们需要引入路由控制模块。

> 其实在那之前我尝试着自己写一个简单的路由控制模块，不过止步于路径参数的解析。\
> 那会儿刚好遇到一些事情，因此也没什么心思造轮子，而事情处理完之后又实在是懒得继续写了，就用了个现成的轮子 `svelte-spa-router`。

#### 关于文章列表页

目前当访问文章列表页时，会去请求 `/src/posts` 下的所有 markdown 文件，并且从加载到的文件中取出元数据，这一方案可以预见的是随着文章数量增多，耗费时间将越来越长。

因此，我也大概思考了一下之后的优化方向：使用 indexedDB 来本地化所有文章数据，并且在每次访问文章列表页时，判断是否有需要更新的文章或者新增的文章，若有，则将这些文章加载下来，然后存入本地数据库中。这样一来，卡顿只会发生在首次访问博客时，之后就不再会出现一次性发起大量请求的场景了。

如何判断是否有需要更新的文章？我想可以通过编写插件，来为每个 markdown 文件计算出相应的 md5，并写入到输出文件夹中，以此来进行对比。每次打开文章列表页时，请求这份文件，然后与本地缓存的描述文件做对比，收集到**md5发生更新的文章**以及**本地缓存中的描述文件里没有的文章**，然后将它们加载到本地。

#### 关于文章页

目前来说，文章页是一个单独的页面，并通过路由获取到对应的id，再根据id去加载对应的由 markdown 编译成的 svelte 组件，最后由 `svelte:component` 渲染到页面的指定插槽中。

我目前在思考的是这个单独的文章页是否是必要的，或许我可以直接为每个文章注册一个路由？

#### 关于我的介绍页

类似于简历，目前有两块，一块是在首页中进行的相关项目展示，另一块是点击右上角导航中的 **关于我** 按钮时，弹出的模态框中。

其中贴上我的 AVATAR 和一些联系方式，以及几句个人介绍，我想这应该就足够了，如果不够的话还能再加——随时都可以。

#### 关于其他子页面

在主要规划之外的子页面，大致如下：

1. 随笔：用于发布一些博客文章之外的内容，比如说小说？
2. 画廊：用于发布一些个人绘画作品。
3. mini应用：用于发布一些个人制作的独立小游戏或者小应用，完成度基本都不会很高的那种。
